# Walkthrough: v0.10.1 → v0.10.2 升版（BREAKING-LITE PATCH）

> **適用**：採用方專案 charter_version 在 `0.10.0` ~ `0.10.1` 範圍、要升到 `0.10.2`
> **核心改動**：commit hook 加 H7 schema-driven 強制必啟集合 binary 攔截
> **預估時間**：5-10 分鐘
> **向下兼容**：⚠️ **BREAKING-LITE PATCH** — 採用方升完若 profile.yaml 漏強制必啟欄位（如缺 F6）、下次 commit 被 H7 擋
>
> **設計動機**：dogfood signal #46（≥ 3 次）+ #31（≥ 5 次）+ #52 候選 累積爆量 — 2026-05-06 公司專案 dbSDK LIVE：兩個 vendor 跑 verify + doctor 同 session 連續 simulated PASS、profile.yaml 漏 F6 都沒抓、user 親手抓。三層雙重防禦（init/doctor/verify spec）對 F6 強制必啟整體 LIVE 失效 → H7 schema-driven binary 兜底。
>
> **為什麼是 BREAKING-LITE 而不是純 PATCH**：對齊 v0.7.0 mislabel 教訓 + user 紀律「**有衝突就代表沒有向下兼容**」（v0.7.3 北極星 framing）。原本能 commit 的 profile（缺 F6）→ 升完不能 commit = 行為衝突 = BREAKING-LITE。

## 3. ⚠️ 檢查 profile.yaml 對齊 `_required.yaml`（關鍵步驟、BREAKING-LITE 觸發點）

讀 charter 端強制必啟集合：

```bash
cat ~/.agentcharter/tools/profiles/_required.yaml
```

當前 v0.10.2 ship **REQ-001-F6**：`parameters.failure-modes.enable_modes` 必含 `F6`（v0.7.0+ standard/strict 強制必啟）。

檢查採用方 profile：

```bash
grep -A1 "enable_modes" agent-commons/_config/profile.yaml
```

**兩種情況**：

### 情況 A：profile 已含 F6 ✅

```yaml
parameters:
  failure-modes:
    enable_modes: ["F1", "F2", "F3", "F4", "F5", "F6"]
```

→ 跳到 Step 4。

### 情況 B：profile 缺 F6 ⚠️（dbSDK case 範本）

```yaml
parameters:
  failure-modes:
    enable_modes: ["F1", "F2", "F3", "F4", "F5"]   # 缺 F6
```

→ **修法**：手動加 `F6`：

```yaml
parameters:
  failure-modes:
    enable_modes: ["F1", "F2", "F3", "F4", "F5", "F6"]   # 補 F6
```

> **為什麼之前漏 F6 不會被抓？** 對應 dogfood signal #46/#31/#52 LIVE — spec 三層雙重防禦（init/doctor/verify）依賴 LLM 自律執行、LLM 跑 doctor / verify 全 simulated PASS、F6 缺從來沒被抓。v0.10.2 H7 binary 攔截不可繞、commit 時強制驗。

## 升版前後對比 — 結構強制升維

### v0.10.1（升版前）

```
F6 強制必啟紀律：
  init-spec Phase 5b CHECK 7 第 4 條  →  依賴 LLM 跑 init 驗
  doctor-spec §3.7 E605               →  依賴 LLM 跑 doctor 驗
  verify-spec §3.2 B002               →  依賴 LLM 跑 verify 驗

實際觀察：兩個 vendor 跑 doctor + verify、F6 缺都沒抓（simulated PASS）
紀律保證強度：弱（單 actor 自律、循環依賴 LLM）
```

### v0.10.2（升版後）

```
F6 強制必啟紀律：
  init-spec Phase 5b CHECK 7 第 4 條  →  advisory（同前）
  doctor-spec §3.7 E605               →  advisory（同前）
  verify-spec §3.2 B002               →  advisory（同前）
  **commit-hook H7 (v0.10.2 加)        →  binary 不可繞攔截 ← NEW**

實際保證：commit 時 H7 binary 強制驗、bypass 須走 --no-verify（user 知情）
紀律保證強度：強（結構強制、四層雙重防禦：spec 三層 advisory + binary 一層兜底）
```

→ 對應 v0.8.2 §設計哲學第 5 條雙軸「**結構強制 > 多 actor 互檢 > 單 actor 自律**」最強保證軸落地。

## 未來 F7 / F8 怎麼解？（schema-driven 紀律）

**charter maintainer 端**：

1. 改 `tools/profiles/_required.yaml` 加新 entry：
   ```yaml
   - id: "REQ-002-F7"
     path: "parameters.failure-modes.enable_modes"
     must_contain: ["F7"]
     since_charter: "0.X.0"
     severity: "ERROR"
     ...
   ```
2. 改 `tools/vendor/commons/charter-commit-checks.sh` 加對應 inline check function（5-10 行 bash）
3. ship + 採用方 git pull → 下次 commit 自動驗

**採用方端**：完全不需要做任何事 — `git pull` + 下次 `install-git-hooks.sh --update` 即傳播。

→ 設計上保持 schema-driven 單一擴展點、避免 hook 數爆炸（不需 H8 / H9 / H10）。對齊 v0.10.0 commit-hook vendor 中立架構「**charter 升新檢項 → user `git pull` → 自動帶最新邏輯**」精神。
