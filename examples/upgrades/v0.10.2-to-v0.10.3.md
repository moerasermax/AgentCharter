## 為什麼這麼簡單？

本 release 屬「**純 spec sweep + maintainer-only lint**」性質：

| 改動類別 | 影響採用方？ |
|---|---|
| `tools/post-upgrade-verify-spec.md` §3.1-§3.5 全升 spec-as-data 四欄結構（22 個 check item）| ❌ 不影響執行邏輯 — verify 工具仍跑同樣的檢項、只是 spec 文件更詳細（讓 LLM 看反例段就知道不能 simulated）|
| `tools/doctor-spec.md` §3 段首加全局「真實 stdout 證據要求」紀律段 | ❌ 不影響執行邏輯 — doctor 仍跑同樣的檢項、只是 spec 寫明 LLM 必貼 stdout |
| `README.md §設計哲學`第 5 條加「升維軌跡」段 | ❌ 純文檔層 framing 擴增 |
| `core/maintainer-discipline.md §3.1.1` 加 lint binary 紀律 | ❌ maintainer-only 紀律、採用方無關 |
| `tools/charter-spec-lint.sh` 新檔（lint binary）| ❌ maintainer-only、採用方不裝不跑 |

→ 所有改動都是「**讓未來 maintainer 加新 spec 不漏結構 + 讓 LLM 跑 verify/doctor 時看反例段降 simulated 率**」、不改採用方執行邏輯。

## 升版前後對比 — spec 結構強制升維

### v0.10.2（升版前）

```
spec 結構合規紀律：
  v0.8.1 ship doctor §3.7-§3.9 四欄結構（5 個 check items）
  v0.8.2 ship 21 條 core/*.md 雙軸 blockquote
  ↑ 部分 spec 對齊、其他 spec 沒對齊（spec drift）

執行紀律：
  LLM 跑 verify / doctor 時靠 LLM 自律真跑、無 stdout 提供反例
  → simulated PASS 率高（dogfood signal #31 LIVE 5 次同類）

maintainer 紀律：
  加新 spec 對齊「該補項」全靠 maintainer 自律
  → 容易漏（dogfood signal #6 / #24 LIVE 多次）
```

### v0.10.3（升版後）

```
spec 結構合規紀律：
  v0.10.3 ship verify §3.1-§3.5 全 5 軸 22 個 check items 四欄結構
  v0.10.3 ship doctor §3 全局「真實 stdout 證據要求」紀律段
  ✅ 全 spec 對齊、零 drift

執行紀律：
  LLM 跑 verify / doctor 時看反例段「❌ 這樣寫」「✅ 正解」
  → simulated PASS 率降（反例段 anti-pattern 顯化）+ commit-hook H7 兜底（v0.10.2 ship）

maintainer 紀律：
  ship 前跑 bash tools/charter-spec-lint.sh
  → L1 偵測 tool spec 缺四欄 / L2 偵測 core 缺雙軸
  → 0 errors 才 ship、結構強制不可繞
```

→ 對應 v0.8.2 §設計哲學第 5 條雙軸「**結構強制升維**」軸落地、charter 完成 SSS S3 propagate 終局。
