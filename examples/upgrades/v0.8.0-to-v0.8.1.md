## 0. 為什麼這個 walkthrough 重要

v0.8.1 是 charter 第一個**純擴增 PATCH 對應 walkthrough**（v0.8.0 是 MINOR 含 BREAKING-LITE、有獨立 walkthrough；v0.7.5 是純擴增 PATCH 但沒寫對應 walkthrough — v0.8.1 補上此設計缺口）。對既有採用方而言：

| 採用方狀態 | v0.8.1 影響 |
|---|---|
| **A. 全合規 + 採用方文檔變更歷史齊**（已有 ADOPTION/TUTORIAL/QUICKSTART 變更歷史段、最新 entry 對齊 charter v0.8.0）| 純擴增升版、零動作 migration（只改 `charter_version` 一行）|
| **B. 採用方文檔變更歷史漏 entry**（如自 fork ADOPTION 但 v0.8.0 entry 沒補）| doctor §3.10 抓 **W901**（警告）；升版前須跑修補 prompt 補 entry |
| **C. 採用方無自寫文檔層**（用 charter 默認 ADOPTION/TUTORIAL/QUICKSTART、未 fork）| W901 不適用（mapping 未指向採用方文檔）；純 charter_version 升 |

→ 本 walkthrough 對 A/B/C 三類採用方分別給 step-by-step 流程 + 每步可貼的 AI prompt。

## 1. 升版前狀態（v0.8.0 baseline 假設）

採用方在 v0.8.0 升版完成（已 follow [v0.7.5-to-v0.8.0.md](./v0.7.5-to-v0.8.0.md)）後的 baseline：

```
<your-project>/
├── agent-commons/                       ← 或採用方覆寫的 common_memory_root
│   ├── _config/
│   │   ├── profile.yaml                 ← charter_version: "0.8.0"
│   │   └── mapping.yaml
```

`profile.yaml` 關鍵欄位：
```yaml
charter_version: "0.8.0"

  failure-modes:
    enable_modes: [F1, F2, F3, F4, F5, F6]   # F6 必啟（v0.7.0 起）
```

## 3. 升版 4 步流程（每步給 AI 的 prompt 模板）

> ⚠️ **signal #21 紀律對齊**：每步都是「**user 動作 = 貼一個 prompt**」、AI 動作 = 完成檔修。**不要求 user 自己編輯 yaml/markdown**（除非 prompt 內明示）。

### Step 1：拉 charter 最新版

**user 動作**：開終端機、跑：

```powershell
cd $env:USERPROFILE/.agentcharter   # PowerShell；macOS/Linux 用 cd ~/.agentcharter
git pull origin main
git log --oneline -3
```

**驗證**：git log 含 commit `0cf5494`（v0.8.1）。

**stdout 範例**：
```
0cf5494 feat(core): v0.8.1 — SSS S3 起手實證 + dogfood signal #24 升工具層 + #19 順手修
afcd330 chore(claude_temp): multi-perspective 第十四循環 + SSS S1 LIVE prototype + 雙軸矩陣議程候選
ab28960 chore(claude_temp): SSS S3 引導式紀律 capture + #25/#26/#27/#28 signal 候選 + 外部 AI 評估第一例
```

---

### Step 2：跑 doctor pre-check（抓 W901）

**user 動作**：切回專案 working dir、貼以下 prompt 給你的 AI（PM Gemini / Engineer Claude 任一皆可）：

```
我要從 charter v0.8.0 升 v0.8.1。請依 ~/.agentcharter/tools/doctor-spec.md
模式 A 對本專案的 agent-commons/ 跑完整健康檢查、特別關注 v0.8.1 新加的校驗：

- §3.10 採用方文檔變更歷史 sync 校驗（v0.8.1 新加、W901）：
  對 ADOPTION.md / TUTORIAL.md / QUICKSTART.md 三採用方文檔（依 mapping.yaml
  shared.adoption_doc / shared.tutorial / shared.quickstart 指向實際檔；若採用方
  未 fork 此三檔、跳過、不報錯）：
  讀檔末段「變更歷史」表 → 比對最新 entry 對應的 charter version vs profile.yaml
  charter_version → 不對應 → W901

完成後請貼出：
(a) doctor 完整 stdout
(b) 若有 W901、列出每項對應的修補建議（不要直接動手修、只列建議）
```

**預期結果分支**：

| 結果 | 你是哪類採用方 | 下一步 |
|---|---|---|
| **0 ERROR / 0 WARN** | A 類（全合規）或 C 類（無自寫文檔層）| 跳到 [Step 4](#step-4升-charter_version) |
| **W901 警告** | B 類（採用方文檔變更歷史漏 entry）| 進 [Step 3.B](#step-3b補採用方文檔變更歷史-entryb-類採用方) |
| 其他 ERROR | 罕見、停下回報 | — |

> 💡 **A vs C 區分**：A 類 = 採用方 fork 了 charter 的 ADOPTION/TUTORIAL/QUICKSTART 並寫了自己的變更歷史對齊；C 類 = 採用方未 fork、直接用 charter 默認檔。doctor 對 C 類自動跳過此校驗、不報錯。

---

### Step 3.B：補採用方文檔變更歷史 entry（B 類採用方）

**user 動作**：貼以下 prompt 給 AI：

```
依 ~/.agentcharter/core/maintainer-discipline.md §3.4.2 文檔層 sync checklist
紀律、補本專案採用方文檔的變更歷史 entry：

對 doctor §3.10 抓到的 W901 每項：
1. 找出對應採用方文檔（ADOPTION.md / TUTORIAL.md / QUICKSTART.md）的「變更歷史」段
2. 在最新 entry 之前（保留歷史 entry append-only）加新 entry：
   - 格式：`v<X.Y>（<YYYY-MM-DD>，charter v<A.B.C>）— <change-summary>`
   - charter version 須對齊 profile.yaml charter_version 目標值（即 v0.8.1）
3. 如果採用方文檔自 v0.8.0 升版時就漏 entry、補完整版本鏈（v0.8.0 + v0.8.1）

不要動既有 entry（保歷史完整）；只在最新 entry 之前 prepend 新 entry。

完成後請：
1. 貼出每個改完檔案的「變更歷史」段 head 5 行
2. 重跑 ~/.agentcharter/tools/doctor-spec.md §3.10 校驗、確認 W901 都消除
3. 結尾貼出 doctor stdout
```

**驗證點**：doctor §3.10 0 WARN。

→ 完成後跳 Step 4。

> 📌 **AI 不可代決升版本身**：W901 是文檔層 sync 漏、不是升版未完成；AI 補變更歷史 entry ≠ 升版完成。`charter_version` 對齊本身在 Step 4 進行（依 profile.yaml charter_version 目標值）。

### Step 4：升 charter_version

**user 動作**：貼以下 prompt 給 AI：

```
請編 agent-commons/_config/profile.yaml（或採用方 common_memory_root 對應位置）：
把 charter_version 從 "0.8.0" 改為 "0.8.1"。只改這一行、其他不動。

完成後貼出：
(a) 改完的 charter_version 該行（grep / Select-String 驗證）
(b) 重跑 ~/.agentcharter/tools/doctor-spec.md 模式 A 完整健康檢查
    確認所有 ERROR/WARN 都消除（含 §3.7 / §3.8 / §3.9 / §3.10）
```

**驗證點**：doctor 全綠 0/0/0。

## 6. 設計學意義

### 6.1 對應 v0.7.3 北極星「不讓 user 記」

v0.8.1 ship 的 dogfood signal #24 升工具層（doctor §3.10 W901）對應同源精神 — **流程強制告訴 user「採用方文檔變更歷史漏 entry」、不依賴 user 跨 release 記得補**。

對應到本 walkthrough：
- 即使 user 連續 release ship 過程沒記得補變更歷史 → 跑 `/charter-doctor` → 流程主動告訴你「W901：ADOPTION.md 變更歷史 entry 不對齊 charter v0.8.1」
- → 你照本 walkthrough Step 3.B 跑修補 prompt → W901 消除 → 真正升版完成

**結構修正 >> 規範補丁**（dogfood signal #22 同源精神 — checklist 是規範補丁、doctor §3.10 W901 是結構修正）。

### 6.2 dogfood signal #21 紀律 LIVE 修正延伸

v0.8.0 walkthrough 已實證「**user 動作 = 貼 prompt**」紀律；v0.8.1 walkthrough 延伸：

| 動作 | 對齊紀律 |
|---|---|
| ❌ 不寫「請手動編 ADOPTION.md 變更歷史段、加 v1.X entry」（user 視角）| 違反 |
| ✅ 寫「**貼以下 prompt 給 AI**：補變更歷史 entry...」（給 AI 的 prompt）| 對齊 |

每個 step 的 user 動作 = 貼 prompt + 看 stdout、AI 動作 = 改檔 + 驗證。

### 6.3 對應「回鍋開發者無痛」北極星 — 純擴增 PATCH 路徑

v0.7.3 README §設計哲學顯化「回鍋開發者無痛」 — 本 walkthrough 對應「**單 PATCH 升版（純擴增）**」階段：

### 6.4 SSS S3 起手 + 第十四循環同步出現的設計密度

v0.8.1 同 release ship：
- **SSS S3 起手**（doctor-spec §3.7-§3.9 全加四欄 spec-as-data） — 對採用方執行邏輯零影響、但對 spec-driven 紀律的 framing 升維（multi-perspective 第十四循環結構師金礦同源）
