## 0. 為什麼這個 walkthrough 重要

v0.8.2 是 charter 第二個**純擴增 PATCH walkthrough**（前一個 v0.8.0-to-v0.8.1.md）— scope 比 v0.8.1 還更小：v0.8.1 引入新 W901 校驗、v0.8.2 **完全無新校驗**、純文檔層擴增。

對既有採用方而言：

| 採用方狀態 | v0.8.2 影響 |
|---|---|
| **任何狀態**（A 全合規 / B 變更歷史漏 / C 無自寫文檔層）| 純擴增升版、零動作 migration（只改 `charter_version` 一行）|

→ v0.8.2 升版步驟是 charter 系列**最簡的 walkthrough**（2 步）— 改一行 + 跑 verify。

### 0.1 「跨版本到最新」自動工作（dogfood signal #29 LIVE 實證、charter 設計成果）

> 📌 **重要設計提示**（user 升公司專案 LIVE 抓到、2026-04-30）：
>
> walkthrough 名稱寫「v0.8.1 → v0.8.2」、但 `git pull origin main` 實際拉的是 **main branch 最新 commit**（可能是 v0.8.2 / 之後 PATCH / 任何最新版）。
>
> 對齊 charter 設計：
> - **v0.5.9 純規範框架**（無 binary、git pull = 拿到最新規範）
> - **v0.7.4 雙軌節奏**（頻繁小擴增 PATCH、每個 PATCH 純擴增向下兼容）
> - **v0.7.0 三層雙重防禦**（doctor 強制偵測累積新校驗）
>
> → 即使你公司專案停在 v0.7.5（甚至更舊）、跑 `git pull origin main` 後直接拿到最新（如 v0.8.2）— **跨版本升級自動工作**、不需按版本順序逐步升。
>
> charter 為這個跨版本場景提供合併路徑指引、見本檔 §5「跨版本場景（停在更舊版）」。

## 3. 升版 2 步流程（每步給 AI 的 prompt 模板）

> ⚠️ **signal #21 紀律對齊**：每步都是「**user 動作 = 貼一個 prompt**」、AI 動作 = 完成檔修。

### Step 1：拉 charter 最新版

**user 動作**：開終端機、跑：

```powershell
cd $env:USERPROFILE/.agentcharter   # PowerShell；macOS/Linux 用 cd ~/.agentcharter
git pull origin main
git log --oneline -3
```

**驗證**：git log 含 commit `a2adecf`（v0.8.2）。

**stdout 範例**：
```
a2adecf feat(core): v0.8.2 — 雙軸矩陣 framing 第一段（multi-perspective 結構師金礦落地）
cb82e11 docs(upgrades): v0.8.0 → v0.8.1 walkthrough 補完 + 採用方文檔層 cross-reference
0cf5494 feat(core): v0.8.1 — SSS S3 起手實證 + dogfood signal #24 升工具層 + #19 順手修
```

> 💡 **跨版本場景**（user LIVE 實證紀律提示）：若你停在 v0.7.5 / v0.7.x / v0.8.0、`git pull` 後 git log 會直接顯示更新的最新 commit（如 v0.8.2）— 這是正常的、charter 跨版本自動向下兼容。直接走本 walkthrough、profile.yaml `charter_version` 改到 `"0.8.2"` 即可。詳見 §5。

---

### Step 2：升 `charter_version` + 跑 `/charter-upgrade-verify`

**user 動作**：貼以下 prompt 給 AI：

```
請編 agent-commons/_config/profile.yaml（或採用方 common_memory_root 對應位置）：
把 charter_version 從 "0.8.1" 改為 "0.8.2"。只改這一行、其他不動。

完成後：
(a) 貼出改完的 charter_version 該行（grep / Select-String 驗證）
(b) 直接打 /charter-upgrade-verify（你已自具象化過此 slash command；spec 不變、charter_version 期望值對齊新版即可）
(c) 貼出 verify report 5 軸 stdout（期望 0 ERROR / 0 WARN / 0 INFO）
```

**預期 stdout**：

```
# AgentCharter Upgrade Verify Report — <project-name>

> Charter version (declared)：0.8.2
> Charter version (actual ~/.agentcharter)：v0.8.2 @ a2adecf
> Profile preset：standard
> 整體狀態：✅ 全綠

[5 軸 A/B/C/D/E 全綠]

📊 **Summary**：✅ 0 ERROR / 0 WARN / 0 INFO
**狀態**：v0.8.2 升版完整、可進工作階段。
```

**驗證點**：5 軸全綠 = v0.8.2 升版完成。

→ **就這 2 步**。v0.8.2 是 charter 升版系列**最無感**的 PATCH。

## 6. 設計學意義

### 6.1 對應 v0.7.3 北極星「回鍋開發者無痛」

v0.8.2 是「**回鍋開發者無痛**」北極星的**極致實證** — charter 升版系列最無感的 PATCH：

| 北極星紀律 | v0.8.2 對齊 |
|---|---|
| 採用方不該記 prompt | ✅ 改一行 + 跑 verify、無修補步驟 |
| user 最少做 1 個動作 | ✅ user 動作 = 貼 1 個 prompt |
| charter 引導採用方、不讓 user 記 | ✅ 升版步驟在 walkthrough 內、AI 跑完報結果 |

### 6.2 對應 v0.7.4 雙軌節奏「頻繁小擴增 PATCH」

v0.8.0（MINOR、含 BREAKING-LITE）→ v0.8.1（PATCH、新 W901）→ v0.8.2（PATCH、無新校驗）三 release 同 24-48 hr 內 ship — 對齊雙軌節奏「頻繁小擴增」精神：每個 release 純擴增、向下兼容、user 升版動作從「修多項 + 升版號」漸減到「純升版號」。

### 6.3 對應「跨版本到最新」設計成果（dogfood signal #29 LIVE 實證）

charter walkthrough 系列名稱寫「vA → vB」、但實際提供「**從 vA 到任何 ≥ vB 的最新版**」自動工作能力 — 來自純規範框架 + 純擴增 PATCH 紀律。

對齊 multi-perspective 評估第十四循環採用方金礦「**跨版本升級自動工作**」精神（無需 user 按版本順序逐步升、直接到最新即可）。

### 6.4 雙軸矩陣 framing 第一段落地（multi-perspective 結構師金礦）

v0.8.2 ship 把雙軸正交分類（物理依據 ⊥ 檢測時點）顯化進 charter 條款體系：

- **物理依據軸**（保證強度）：結構強制（強）/ 多 actor 互檢（中）/ 單 actor 自律（弱）
- **檢測時點軸**：init / runtime / post-upgrade / handoff

→ 採用方升版完 v0.8.2 後、可讀 [`README.md §設計哲學`](../../README.md) 第 5 條了解此 framing — 對主動加固專案弱保證項有幫助。
