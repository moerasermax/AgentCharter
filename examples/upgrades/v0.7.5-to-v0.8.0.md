## 0. 為什麼這個 walkthrough 重要

v0.8.0 是 charter 第一個**含可接受 BREAKING-LITE 的 MINOR**（doctor 校驗強化 — §3.8 vendor schema + §3.9 axiom status 從 spec 層升實作層）。對既有採用方而言：

| 採用方狀態 | v0.8.0 影響 |
|---|---|
| **A. 全合規**（v0.7.5 接入時 Gemini toml 扁平 + axiom USER-RATIFIED）| 純擴增升版、零動作 migration（只改 `charter_version` 一行）|
| **B. vendor schema 不合規**（如 toml 含 nested table）| doctor §3.8 抓 E801；升版前須跑修補 prompt |
| **C. axiom 未升 USER-RATIFIED**（路徑 B 走過但 user 未升 status）| doctor §3.9 抓 E606；升版前須跑校 axiom + 升 status prompt |

→ 本 walkthrough 對 A/B/C 三類採用方分別給 step-by-step 流程 + 每步可貼的 AI prompt。

## 3. 升版 5 步流程（每步給 AI 的 prompt 模板）

> ⚠️ **signal #21 紀律對齊**：每步都是「**user 動作 = 貼一個 prompt**」、AI 動作 = 完成檔修。**不要求 user 自己編輯 yaml/markdown**（除非 prompt 內明示）。

### Step 1：拉 charter 最新版

**user 動作**：開終端機、跑：

```powershell
cd $env:USERPROFILE/.agentcharter   # PowerShell；macOS/Linux 用 cd ~/.agentcharter
git pull origin main
git log --oneline -3
```

**驗證**：git log 含 commit `b5866b5`（v0.8.0）或更新（如 v0.8.1 `0cf5494` / v0.8.2 `a2adecf` / 之後）+ tag `v0.8.0`。

> 💡 **跨版本場景紀律提示**（dogfood signal #29 LIVE 實證、2026-04-30 user 升公司專案抓到）：`git pull origin main` 拉 main branch **最新 commit**、可能 ≥ 本 walkthrough 名稱版本（v0.8.0）— **這是正常的、charter 跨版本自動向下兼容**。
>
> 直接走本 walkthrough 即可、最後升 `charter_version` 時可**直接到最新版**（如 `"0.8.2"` 而非中間版本）— 不需按版本順序逐步升。
>
> 詳細跨版本路徑：[v0.8.0-to-v0.8.1.md §5](./v0.8.0-to-v0.8.1.md#5-v075-直跳-v081-場景合併兩段升) + [v0.8.1-to-v0.8.2.md §5](./v0.8.1-to-v0.8.2.md#5-跨版本場景停在更舊版本)。對齊 charter 純規範框架 + 純擴增 PATCH 紀律 + v0.7.3 北極星「回鍋開發者無痛」。

**stdout 範例**：
```
b5866b5 feat(core): v0.8.0 — 升版 + 接入防呆強化 / signal #23 條款化 + #16 升實作層 + #10 升結構修正
eb4dd46 docs(adopters): v0.8.0 採用方文檔變更歷史補完 + signal #24 候選紀錄
fdd77a7 chore(claude_temp): 記入 v0.7.6 BOOTSTRAP 議程備註 — 必含升版快速執行版
```

### Step 2：跑 doctor pre-check（抓 v0.8.0 新校驗會踩的問題）

**user 動作**：切回專案 working dir、貼以下 prompt 給你的 AI（PM Gemini / Engineer Claude 任一皆可）：

```
我要從 charter v0.7.5 升 v0.8.0。請依 ~/.agentcharter/tools/doctor-spec.md
模式 A 對本專案的 agent-commons/ 跑完整健康檢查、特別關注 v0.8.0 新加 / 啟用的校驗：

- §3.8 vendor schema check（v0.8.0 啟用 E801/W802）：
  對 .gemini/commands/*.toml 或 .claude/commands/*.md 校驗
  vendor schema 規範段（roles/pm/gemini-cli.md §3.6 + roles/engineer/claude-code.md §4.1）

- §3.9 axiom 紀律對齊（v0.8.0 新加 E606/E607/W608）：
  對 agent-commons/protocols/<axiom>.md frontmatter 校驗
  status 必須為 USER-RATIFIED；缺 mutability_default 報 W608

完成後請貼出：
(a) doctor 完整 stdout
(b) 若有 ERROR/WARN、列出每項對應的修補建議（不要直接動手修、只列建議）
```

**預期結果分支**：

| 結果 | 你是哪類採用方 | 下一步 |
|---|---|---|
| **0 ERROR / 0 WARN** | A 類（全合規） | 跳到 [Step 4](#step-4升-charter_version) |
| **E801 / W802** | B 類（vendor schema 不合規） | 進 [Step 3.B](#step-3b修-vendor-schema) |
| **E606 / E607 / W608** | C 類（axiom 未升 status） | 進 [Step 3.C](#step-3c升-axiom-status) |
| 其他 ERROR | 罕見、停下回報 | — |

### Step 3.B：修 vendor schema（B 類採用方）

**user 動作**：貼以下 prompt 給 vendor AI（Gemini for toml / Claude for md）：

```
依 ~/.agentcharter/roles/pm/gemini-cli.md §3.6（Gemini CLI toml 扁平結構強制）
[或 ~/.agentcharter/roles/engineer/claude-code.md §4.1（Claude Code .md 純 markdown 規範）]
修補本專案 .gemini/commands/*.toml [或 .claude/commands/*.md]：

- 移除 [command] / [command.instruction] 等 nested table
- 移除 name 欄位（檔名 = 指令名）
- description + prompt 直接寫在 root level
- 純 markdown（.md）/ 純扁平 toml（.toml）

完成後請：
1. 貼出每個改完檔案的 head 15 行（驗證結構合規）
2. 重跑 ~/.agentcharter/tools/doctor-spec.md §3.8 校驗、確認 E801/W802 都消除
3. 結尾貼出 doctor stdout
```

**驗證點**：doctor §3.8 0 ERROR / 0 WARN。

→ 完成後跳 Step 4。

### Step 3.C：升 axiom status（C 類採用方）

**注意**：本步**user 須親自決策 axiom 內容**（dogfood signal #5「LLM 不可繞過 user 決策」+ multi-role-tracking §3.4.4「升 status 必須 user explicit 授權」）。**AI 不可代升 status**。

#### 3.C.1 user 親自校 axiom

**user 動作**：開 `agent-commons/protocols/<axiom>.md`、依 [`templates/agent-commons/domain-axioms-via-ai-draft-prompt.md.tpl §user 校正 checklist`](../../templates/agent-commons/domain-axioms-via-ai-draft-prompt.md.tpl) 跑 6 條 checklist：

- [ ] **真實性**：每條鐵律真的來自 codebase 而非編造？
- [ ] **適用性**：對未來 6 個月仍適用？
- [ ] **可驗證性**：「驗證」段給的方法能跑通？
- [ ] **後果段量級**：「不只是 bug、是事故」（資金 / PII / 合規 / 服務中斷）？非「會出錯」這類弱描述？
- [ ] **編號**：①②③ 連續 + 不重複？
- [ ] **缺漏**：心裡有的「事故」場景 AI 是否漏抓？親自補

決定每條鐵律：**保留 / 改寫 / 降為 architectural guideline / 刪**。記下 N 條保留 / M 條改寫 / K 條刪除 / L 條新增。

#### 3.C.2 user 貼 prompt 讓 AI 改 frontmatter

**user 動作**：貼以下 prompt 給 AI：

```
我已校完 agent-commons/protocols/<axiom>.md 內容（決定：N 條保留 / M 條改寫 / K 條刪除 / L 條新增）。
請依 v0.7.1 路徑 B 紀律 + dogfood signal #23 條款化執行載體（doctor §3.9）
更新本檔 frontmatter：

- status: AI-DRAFTED → USER-RATIFIED
- created_by: ai-drafted → user-ratified-from-ai-draft
- 加 ratified_at: <今天日期 YYYY-MM-DD>
- 在 frontmatter 後 + 標題前加一行：
  > 校正紀錄：<YYYY-MM-DD> user 校正完成、N 條保留 / M 條改寫 / K 條刪除 / L 條新增

不要動 axiom 內容（鐵律本身）— 那是我已校完的、純改 frontmatter + 加校正紀錄行。

完成後貼出：
(a) 改完的 head 15 行（看 frontmatter + 校正紀錄行）
(b) 重跑 doctor §3.9 確認 E606/E607/W608 都消除
```

**驗證點**：doctor §3.9 0 ERROR / 0 WARN。

→ 完成後跳 Step 4。

## 6. 設計學意義

### 6.1 對應 v0.7.3 北極星「不讓 user 記」

v0.8.0 ship 三層雙重防禦（init-spec Phase 5b CHECK 7 ext + doctor §3.9 + post-upgrade-verify 軸 D）的同源精神 —— **流程強制告訴 user 修補、不依賴 user 跨 session 記得修**。

對應到本 walkthrough：
- 即使 user 下班 → 明天回來 → 跑 `/charter-upgrade-verify` → 流程主動告訴你「軸 D D001 ERROR：axiom AI-DRAFTED」
- → 你照本 walkthrough Step 3.C 跑修補 prompt → 5 軸全綠 → 真正升版完成

**結構修正 >> 規範補丁**（dogfood signal #22 同源精神）。

### 6.2 dogfood signal #21 紀律 LIVE 修正

**user 2026-04-29 抓到的設計矛盾**：「**所有採用方需要動 schema / 修檔的動作 — 必須有對應給 AI 的 prompt 模板、不可只給 user 視角的 diff 範例**」。

本 walkthrough 嚴格對齊：
- ❌ 不寫「請編 profile.yaml、把 charter_version 改為 0.8.0」（user 視角）
- ✅ 寫「**貼以下 prompt 給 AI**：請編 profile.yaml、把 charter_version 改為 0.8.0」（給 AI 的 prompt）

每個 step 的 user 動作 = 貼 prompt + 看 stdout、AI 動作 = 改檔 + 驗證。
