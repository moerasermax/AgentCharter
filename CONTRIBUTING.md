# Contributing to AgentCharter

> 歡迎不同 AI / 不同專案 / 不同角色的維護者貢獻。

---

## 1. 歡迎的貢獻類型

| 類型 | 說明 |
|---|---|
| 新 AI 實作（`roles/<role>/<ai-vendor>.md`）| 你代表（或你就是）某個 AI，提交該 AI 在某個角色上的執行細節 |
| 新角色（`roles/<role>/`）| 提出新職能（reviewer、qa、sre…），含 `_spec.md` + 至少一個 AI 實作 |
| 新失敗模式（`core/failure-modes.md` 內擴增）| 真實事件觸發的新偏差類型 |
| 新核心條款（`core/*.md`）| 跨 AI / 跨專案 / 跨角色都適用的通用原則 |
| 新範例（`examples/<project>/mapping.md`）| 你的專案採用 AgentCharter 的對照表 |
| 修訂 / 補強現有條款 | PR 即可 |

---

## 2. PR 須附

| 項目 | 必含 |
|---|---|
| 變更摘要 | ✅ |
| 動機（為何這個改動）| ✅ |
| 真實事件背書（如新失敗模式 / 新核心條款）| ✅ |
| 與既有條款的關係 / 差異說明 | ✅ |
| 影響範圍評估（哪些 AI / 角色 / example 會被影響）| ✅ |

---

## 3. 提交流程

依 `GOVERNANCE.md §2` 對應的變更類型走：

1. fork repo
2. 建 branch
3. 寫變更
4. 開 PR，照 PR template 填
5. 邀請對應的 reviewer（依 GOVERNANCE 規定的人數）
6. 審核通過 merge

---

## 4. 跨 AI 提交特別說明

如果你**就是**那個 AI（例如 Gemini CLI 自己代表自己提交）：

- 在 PR 描述中明示「本 PR 由 <AI 名稱> 代理提交，背後人類維護者：<姓名 / handle>」
- 提交內容應反映該 AI 的真實能力與行為，避免「廠商宣傳口吻」
- 已知盲區段落（依 spec §3 必含）必須誠實，不得隱瞞

---

## 5. 命名與格式

| 規範 | 細節 |
|---|---|
| 檔名 | 全小寫 + `kebab-case`（如 `audit-rights.md`，不用 `auditRights.md`）|
| AI 廠商標識 | 用通用識別字串（`claude-code` / `gemini-cli` / `cursor`），避免內部代號 |
| 條款編號 | 用 `§` 前綴；引用其他條款用相對路徑（如 `audit-rights.md §2`）|
| 條款語氣 | 中文 + 英文小標題並陳；國際化支援由社群分擔 |

---

## 6. 不歡迎的內容

| 類型 | 為何不歡迎 |
|---|---|
| 廠商宣傳 | 框架是公共財，不是廣告位 |
| 未經實證的條款（無真實事件背書）| 違反 `evidence-first.md` 原則 |
| 對特定 AI / 廠商的攻擊 | 框架旨在多 AI 協作，不是站隊 |
| 抄襲未授權內容 | LICENSE 議題；待 v1.0 公開化後明確化 |

---

## 7. 行為準則

- 對其他貢獻者保持禮貌
- 同一條款的爭議用條款理由說服，不用人身或廠商立場
- 失敗模式的紀錄是學習資產，不是恥辱牆 — 真實提交比掩蓋珍貴

---

## 8. v0.x 階段的特別說明

當前框架仍在 v0.x，治理機制尚未完整啟動。重大變更暫由創始者直接審核 + 至少一名 active 維護者背書即可 merge。v1.0 後切換到 `GOVERNANCE.md` 完整流程。
