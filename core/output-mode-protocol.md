# Output Mode Protocol（輸出模式協議）

> **狀態**：v0.1
> **位階**：core 通用條款。
> **保證強度**：單 actor 自律（模式切換、vendor-specific）
> **檢測時點**：runtime
> **since**：v0.1

---

## 1. 條文

每個 AI 角色的對話輸出受**雙段式模式**控制：

| 模式 | 名稱 | 適用 |
|---|---|---|
| `verbose` | 詳細模式（預設）| 複雜任務、安全關鍵變更、新人交接、首次接觸新領域 |
| `eco` | 節能模式 | 例行修補、查詢、單純驗證 |

模式由專案內**狀態檔**（建議 `<project>/state/output_mode` 或 `<project>/management/power_mode.txt`）決定，所有 AI 角色共用同一旗標檔，避免認知不一致。

---

## 2. eco 模式 — 可削減項目

eco 啟用後角色輸出主動省略：

1. 「為什麼這樣做」之說明性段落（非鐵律引述）
2. 進度旁白、開場 / 結尾的禮貌性鋪陳
3. 多餘的 markdown 標題層級與 emoji 裝飾
4. 重複的確認句
5. 主動補充的延伸說明（除非使用者請求）

---

## 3. eco 模式 — 禁砍項目（即使 eco 也保留）

以下為**安全護欄**或**協作觸發信號**，eco 仍須完整保留：

1. **Directive Header**（如 `📨 致 PM`）— 拿掉協作循環會卡住
2. **領域安全公理引述**（金融專案 = IRON、醫療 = HIPAA…）— 同安全護欄等級
3. **危險度標籤**（📖 唯讀 / ⚠️ 會建立紀錄 / 🔥 會動到資金 / 等對應領域標）
4. **完工交付的期望錨點與失敗解讀表**（依 `completion-delivery.md`）
5. **失敗模式編號 / 條款引述**（F1〜Fn、§編號）— 審計痕跡

---

## 4. 自動升級 — eco → verbose

即使旗標為 eco，遇下列任一條件**自動切回 verbose**（不論旗標）：

- 任務涉及領域安全公理（金融資金、醫療隱私、軍工授權…）
- 任務涉及高危動作（生產環境寫入、實際下單、不可逆操作）
- 使用者明示「請詳細」「展開」「為什麼」「完整紀錄」等關鍵詞

---

## 5. 切換指令

提供與專案 slash command 整合的切換介面（建議格式）：

```
/<mode-cmd> [eco | verbose | status | toggle]
```

範例：CryptoBot 用 `/power [eco|verbose|status|config|toggle]`。

---

## 6. 跨 AI 同步

不同 AI 對狀態檔的處理能力不同：

| 能力 | 應對 |
|---|---|
| 有 hook 系統（如 Claude Code 的 UserPromptSubmit）| 自動每輪注入規範至 context |
| 無 hook 系統（如某些 Gemini CLI 配置）| 對應角色須在「值機初始化」時主動讀檔 + 在輸出前自我提醒 |
| 完全無檔案系統存取 | 由協作另一方在每輪訊息前綴註明當前模式 |

---

## 7. 與其他 core 條款的關係

| 條款 | 關係 |
|---|---|
| `init-template.md` | 角色 init 時必含「讀模式狀態 + 套用對應規範」 |
| `completion-delivery.md` | eco 禁砍清單與完工交付規範重疊保護 |
