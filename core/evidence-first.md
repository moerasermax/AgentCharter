# Evidence-First Principle（實證先行原則）

> **狀態**：v0.1
> **位階**：core 通用條款。
> **保證強度**：單 actor 自律（隱性 bug 嚴禁盲猜、依賴 LLM 紀律）
> **檢測時點**：runtime
> **since**：v0.1

---

## 1. 條文

任何角色在以下情境**嚴禁盲猜**：

| 情境 | 嚴禁 | 應該 |
|---|---|---|
| 隱性 Bug 修復 | 直接改程式碼 | 先寫診斷指令 / 重現步驟，確診後動工 |
| 外部 API 行為（錯碼、訊息、限制）| 用「應該是」「通常會」 | 先用 probe / 文件 / 真實呼叫取證 |
| 效能 / 延遲 / 吞吐 / 規模 | 寫進文件當事實 | 先實測，附 log / metric 截錄 |
| 跨機 / 跨平台行為 | 假設一致 | 在目標環境親跑驗證 |
| 時間 / 數字 / 統計 | 心算 | 用工具（sqlite3、jq、awk、瀏覽器 dev tools）算 |

---

## 2. 設計動機

盲猜的後果累積：

- 一次假設變傳說
- 傳說變協議
- 協議變鐵律
- 鐵律與真實系統脫鉤
- 系統第一次出真事故時集體失能

實證先行是**抗熵**機制：強迫每個寫入文件 / 程式碼的事實，都有可重現的證據鏈。

---

## 3. 實作要點

### 3.1 角色責任分配

| 角色 | 對應實證義務 |
|---|---|
| 任何發起方（含 PM、Engineer、Reviewer） | 對自己提出的「事實型陳述」負責附證據；無證據即標明為假設 |
| 抽驗方 | 拒絕接受無證據的事實陳述（依 `audit-rights.md`）|
| 使用者裁決時 | 可指定「免證據通過」，但須明示 |

### 3.2 證據格式建議

- **檔案存在類**：`ls -la <path>` 原文
- **內容類**：`grep -n` / `cat` 節錄 + 行號
- **行為類**：probe 指令 + stdout 原文
- **數值類**：query / API call + 回傳 raw payload
- **時間類**：明示時區（避免 UTC ↔ 本地誤判，CryptoBot S70 事件即為此案例）

### 3.3 「假設」標籤

當證據暫時不可得（成本高、時間窗口受限），可暫以假設推進，但須**顯式標記**：

```markdown
> ⚠️ ASSUMPTION（待驗）：本段認定 BingX errorCode=101400 對應 duplicate
> client order id。實證來源：S66-A T0 探針已驗。
```

未標記為假設的內容即視為事實宣告，須有證據。

---

## 4. 反捏造原則（Anti-Fabrication）

針對特別容易捏造的類別：

- **不用心算時間 / 數字** — 用工具算
- **不用「記憶中是」** — 重新驗證
- **不用「常見的做法是」** — 查 spec / 跑驗證
- **不用「之前的 session 講過」** — 跨 session 不可繼承事實，須重新 ground

## 5. 與其他 core 條款的關係

| 條款 | 關係 |
|---|---|
| `audit-rights.md` | 抽驗的標的就是「證據鏈」 |
| `failure-modes.md` | F3（捏造數據）即違反本原則 |
| `completion-delivery.md` | 完工交付的 VCP 必含具體驗證指令 + 期望錨點 |
| `structural-anti-fabrication.md` | 證據必須在文檔結構內顯式（stdout 區塊），非純文字描述 |
| `domain-axiom-slot.md` | 領域公理條款內容須可驗證（§3.1）— 與本原則同源 |
