# Audit Rights（抽驗權）

> **狀態**：v0.1
> **位階**：core 通用條款。
> **保證強度**：多 actor 互檢
> **檢測時點**：runtime
> **since**：v0.1

---

## 1. 條文

任一角色發出的「**已完成 / 已建立 / 已落實 / 已校準 / 已更新**」型宣告，**默認狀態為「待抽驗」**（pending audit），須由對應抽驗方以可重現手段驗證後才生效。

> **抽驗權不得放棄**。對抽驗方而言，跳過抽驗就是把責任轉給未來 session 的自己或下一個接班人。

---

## 2. 抽驗手段（依宣告類型對應）

| 宣告類型 | 抽驗指令 |
|---|---|
| 檔案存在 / 重命名 / 刪除 | `ls -la <path>` 證實實體 + 必要時讀檔內容 |
| Git commit hash | `git log --oneline -1 <hash>` 比對訊息與時間戳 |
| 段落 / 章節寫入 | `grep -c "<關鍵字>" <file>` 或讀檔對照 |
| 測試結果 | 親跑 `dotnet test` / `pytest` / 對應 test runner |
| 數值 / 統計 | 親跑 `sqlite3` / `psql` / `curl` 對應 query 或 endpoint |
| 行號 / 章節編號 | `grep -n` 比對實際行號 |
| 外部系統狀態 | 親跑 probe（如 BingX `probe-bingx`、AWS `aws sts get-caller-identity`）|

「相信對方說的」**不是合法抽驗**。

---

## 3. 抽驗 SOP（標準作業程序）

1. **接收**：抽驗方收到結案宣告，**不得直接信任**
2. **驗證**：依 §2 對應指令親跑，留下原文紀錄
3. **判決**：
   - 全綠 → 發出明確「✅ 抽驗通過」字樣
   - 任一項紅 → 立即退稿，列出**具體條目**（不得只回「有問題」）
4. **退稿循環**：宣告方補強 → 抽驗方重新抽驗
5. **生效**：抽驗通過後，閉環才正式生效

---

## 4. 信任邊界揭示

當完整 stdout 原文無法取得（跨 session、跨機器、跨 AI 邊界），抽驗方須在歷史紀錄中**透明標註信任邊界**：

```markdown
### 信任邊界揭示

- 抽驗項 X 由 PM 提供摘要結論，未取得 stdout 原文
- 抽驗方依 audit-rights §2 採可獨立驗證手段（sqlite3 / grep / 測試）做替代抽驗
- 所有可獨立驗證項目對齊，故採信本輪宣告
```

不可隱藏「我沒驗到原文」的事實。

---

## 5. 例外條款

唯一可跳過抽驗的場景：

- **使用者明確下達**「直接放行」「不需抽驗」「我自己驗過了」等明示指令
- 此時抽驗責任歸使用者，抽驗方退出該閉環，但須在歷史紀錄留下「使用者裁決放行」字樣

「節奏緊湊」「對方看似可信」「之前都對」**皆非合法跳過理由**。
