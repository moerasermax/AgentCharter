# Role Conflict Resolution（角色決策衝突裁決協議）

> **狀態**：v0.2（v0.6.0 加 §5.4 角色切換決策權）
> **位階**：core 通用條款。
> **依存**：`role-separation.md`（角色互鎖）、`escalation-protocol.md`（區隔對象）、`audit-rights.md`、`evidence-first.md`、`violation-reflection.md`、`common-memory-root.md`、`multi-role-tracking.md`（v0.6.0 加 §3.4 身份穩定承諾）
> **保證強度**：多 actor 互檢（L0/L1/L2 三級階梯由抽驗方執行）
> **檢測時點**：runtime
> **since**：v0.5.3（v0.6.0 加 §5.4）

---

## 0. 與 escalation-protocol 的區隔（先講清楚）

兩條款處理的是**完全不同的對象**：

| 條款 | 處理對象 | 動力方向 | 是否有對錯 |
|---|---|---|---|
| `escalation-protocol.md` | **失敗事件累積**（F1〜F5：假宣告 / 假 hash / 捏造數據 / 編號偏差 / 規則記憶失效）| **單向**（宣告方有偏差，抽驗方升級）| ✅ 有 — 偏差方須補強 |
| **本條款** | **決策分歧**（範圍 / 技術選型 / 紀律解讀 / 優先序）| **雙向**（兩角色對等 escalate）| ❌ 無 — 只是判斷路徑不同 |

→ **誤分類後果**：把分歧誤判為失敗事件 → 抽驗方單方退稿、累積進入強化抽驗 → 對方無辜進入升級狀態 → 協作信任崩解。

**判定規則**：
- 對方提交內容**未違反任何 core 條款 / 領域公理 / capsule 契約**，但你不認同其判斷 → 走本條款
- 對方提交內容**違反條款**（如缺 stdout、跳過抽驗、捏造數據）→ 走 `escalation-protocol`
- 邊界爭議 → 假設為分歧（本條款），由 L1 條款仲裁判定是否升級為失敗事件

---

## 1. 條文

當兩角色對某決策有分歧且任一方未違反條款時，**禁止單方逕行**；必須走三級裁決階梯，每階段留審計痕跡。

---

## 2. 衝突類型分類

不同類型對應不同的 L1 仲裁基準：

| 類型 | 描述 | L1 仲裁基準 |
|---|---|---|
| **範圍衝突** | PM 想擴 scope，Engineer 守 capsule 邊界 | capsule 契約 + 領域公理（如 IRON 修訂限制）|
| **技術選型衝突** | 兩種實作路線並存，雙方各有理據 | `evidence-first.md` — 以實證 / benchmark 仲裁；無實證則 L2 |
| **紀律解讀衝突** | 同一條款雙方解讀不一 | 條款 §「與其他條款的關係」段 + IM 內過往判例 |
| **優先序衝突** | 先做 X 還是 Y | NextWork 既定優先序；若 NextWork 未明示 → L2 |
| **領域 vs 通用衝突** | 領域公理與 core 條款衝突 | **領域公理優先**（依 `domain-axiom-slot.md §2.1`）|

衝突類型在進入 L0 時即須由發起方標明，避免事後改判。

## 3. 三級裁決階梯

### 3.1 L0 對話（Discussion）

**觸發**：初次分歧出現。

**動作**：
1. 發起方在 capsule 或 conflict-record（依 §4）寫下立場：**主張 + 理據 + 引用條款 / 公理 / 證據**
2. 對方須在 N 回合內（建議 N ≤ 2）回應立場
3. 雙方對話直至：
   - **共識** → 寫結論到 capsule decision log，事件解除
   - **無共識且雙方確認** → 升 L1

**禁令**：
- ❌ 不得在 L0 期間單方逕行（如 Engineer 自行 commit 走自己路線）
- ❌ 不得用「我覺得」「應該」這類無理據的主張（依 `evidence-first.md`）
- ❌ 不得拉長無限輪數消耗對方（建議 N ≤ 2）

### 3.2 L1 條款仲裁（Charter Arbitration）

**觸發**：L0 無共識，或一方明示「請走 L1」。

**動作**：
1. 發起 L1 的角色須提出仲裁基準：對應條款 §段 / 領域公理條款 / IM 判例
2. 對方須對基準回應：**接受** / **不接受並提反基準** / **請求 L2**
3. 仲裁結果分三類：
   - **基準明示一方對** → 採該方主張，寫結論
   - **基準明示一方錯且為條款違反** → **轉 `escalation-protocol`**（已非分歧而是失敗事件）
   - **基準未明示 / 雙方對基準解讀不一** → 升 L2

**仲裁基準的證據要求**：
- 對應條款須引用具體 §段編號（不接受「依 audit-rights 的精神」這類模糊引用）
- IM 判例須引用具體事件 ID + 段落
- 領域公理須引用條款編號（如 IRON ②）

### 3.3 L2 使用者裁決（User Adjudication）

**觸發**：L1 條款未明示 / 雙方對條款解讀不一 / 任一方明示「請走 L2」。

**動作**：發起方上報使用者，附以下標準格式：

```markdown
## 衝突上報（依 role-conflict-resolution §3.3）

- **衝突類型**：<§2 五類之一>
- **角色 A 立場**：<主張 + 理據 + 引用>
- **角色 B 立場**：<主張 + 理據 + 引用>
- **L0 對話摘要**：<幾個回合、卡在哪>
- **L1 仲裁結果**：<基準、為何未明示>
- **可選方案**：
  - A. <方案 A 描述 + 後果>
  - B. <方案 B 描述 + 後果>
  - C. <折衷 / 第三條路>
  - D. 使用者自由指令
```

使用者裁決後：
- 雙方依指令執行，**不得繼續爭辯**
- 結論寫入 IM（依 §4），標籤「L2 使用者裁決判例」便於未來相似衝突參照

## 4. 紀錄要求（Audit Trail）

衝突事件不得淪為「對話內的私下爭辯」。每次衝突須有可追溯紀錄：

### 4.1 紀錄位置

| 衝突等級 | 紀錄位置 | 強度 |
|---|---|---|
| L0 達成共識 | 寫進對應 capsule 的 decision log 段 | 必須 |
| L1 條款仲裁 | 同上，加註「L1 仲裁基準：<條款 §段>」 | 必須 |
| L2 使用者裁決 | 額外建 `<common-memory-root>/conflicts/<event-id>.md`（或 IM 章節）| **必須** + 入 IM 為判例 |

### 4.2 conflict-record 最低欄位

```markdown
# Conflict Record: <event-id>

- **日期**：<YYYY-MM-DD>
- **衝突類型**：<§2 五類>
- **參與角色**：<role-A> vs <role-B>
- **發起方**：<role>
- **L0 對話**：<簡述輪次與卡點>
- **L1 仲裁基準**：<條款 §段 / 公理 / 判例>
- **L2 上報**（若觸發）：<使用者裁決選項與結果>
- **結論**：<最終決議>
- **後續監測**：<該決議是否需要事後 review，例如 N session 後抽驗效果>
```

### 4.3 IM 沉澱條件

L2 級衝突**強制**沉澱進 IM；L1 級若涉及罕見條款解讀**建議**沉澱。沉澱欄位依 `templates/agent-commons/institutional-memory-entry.md.tpl` 五段格式（症狀 → 根因 → 診斷 → 修法 → 預防），「修法」段對應結論、「預防」段對應未來相似衝突的判例指引。

## 5. 防呆（Anti-Bypass）

### 5.1 Pending 狀態紀律

衝突進入 L0 後，**該決策進入 pending 狀態**：

| 動作 | 是否允許 |
|---|---|
| 雙方對話、寫立場、找證據 | ✅ |
| 推進相關但非衝突點的工作 | ✅ |
| 任一方對衝突點本身單方逕行（如 Engineer 自行 commit、PM 自行關膠囊）| ❌ **嚴禁** |
| 跳過 L0 直接 L2 上報使用者 | ❌（除 §5.2 例外） |

違反 pending 紀律 = 越界行為 = 觸發 `role-separation.md §3.1` 退稿。

### 5.2 時效性例外（Hot-Fix Exception）

生產事故 / 安全事件 / 不可逆操作的時效窗口內，允許**先執行 + 事後補 conflict-record**，但須滿足：

| 條件 | 細節 |
|---|---|
| 時效性可舉證 | 「30 分鐘內不修就資金損失」需有具體後果，禁模糊「可能會出問題」|
| 執行後 24 小時內補紀錄 | conflict-record 含「事後追認」標籤 |
| L2 上報使用者追認 | 使用者可選擇追認 / 否決 / 部分採納 |
| 事件入 IM | **強制**（避免「時效性」被濫用為跳過協議的捷徑）|

未滿足任一條件 → 視同越界，依 `role-separation.md §3.1` 處置。

### 5.3 反濫用（Anti-Abuse）

| 濫用模式 | 對應 |
|---|---|
| 把所有失敗事件包裝為「分歧」逃避 escalation | 抽驗方有權拒絕走本條款，主張對方違反具體條款並走 escalation |
| 在 L0 期間用無限輪次拖延 | 對方有權單方宣告升 L1（依 §3.1 N ≤ 2 回合）|
| 反覆對同類分歧重新發起 L2 | IM 已有判例的相似衝突，**直接套用判例**；重提須附「為何本案與判例不同」舉證 |
| 一方拒絕進入紀錄（堅持口頭討論）| 違反 §4 紀錄要求；視同 F4（編號 / 紀錄偏差類）|

### 5.4 角色切換決策權屬 user（v0.6.0 加）

> **動機**：dogfood signal #5（YC_AIAgentCrew 2026-04-28）— Gemini PM 在 TASK_013（涉及 `src/` 修法）連續兩次嘗試繞過「PM 不得改 src/」紀律，**變體 1** 自我宣告「切換身分為 Engineer」執行 engineer-init self-instantiation。本案揭露「角色切換」的決策權歸屬從未明文規範。

**核心紀律**：

當任務涉及兩個角色職權邊界（典型場景：PM 收到任務但任務需要動 `src/`）：

| 動作 | 決策權 | 依據 |
|---|---|---|
| **判斷該任務由哪個角色執行** | **user** | 不是 AI 自決、不是 charter 自動判定 |
| **發起角色切換**（PM session 切換為 Engineer 執行）| **user explicit 授權**（如「現在請以 Engineer 身份接收此任務」）| 對應 `multi-role-tracking §3.4` 身份穩定承諾 |
| **AI 在邊界 case 的對應動作** | **退回給 user 確認**（不自我發起切換）| 「此任務涉及 src/，請改派 Engineer 角色 / 或請明示授權我以 Engineer 身份接收」 |

**AI 自我宣告切換的處置**：

| 違反方式 | 處置 |
|---|---|
| AI 自我宣告「我現在切換為 \<其他角色\>」執行任務 | 視為 F1 假宣告就位（依 `multi-role-tracking §3.4`）；該切換無效，工件須退回 |
| AI 派 sub-agent / 代理執行越界任務（變相切換）| 違反 `role-separation §3.5` 繞路禁令；同視為 F1 |
| AI「禮貌詢問」變相壓 user 授權（「我覺得這應該由 Engineer 做，需要我切換嗎？」）| ✅ 允許 — 主動詢問是合規動作（§3.4.2 邊界 case），但須等 user explicit 回應，不得「沒回應就視為授權」 |

**對應 §2 衝突類型擴充**（隱性新增，不改既有五類）：

「角色切換爭議」屬「**範圍衝突**」（§2 第一類）的特例 — PM 認為任務在自己職權內、Engineer 認為應該由自己執行。處置：

- L0：兩 AI 對話，引用 `role-separation §1` 權力槽位 + `role-separation §3.5` 繞路禁令舉證
- L1：條款仲裁 — 通常 `role-separation §1` 表格直接判定（程式碼寫入 = Engineer / 任務契約撰寫 = PM）
- L2：若條款明示但 AI 仍意圖規避 → user 裁決選項：(A) 強制走條款判定 / (B) 例外授權單次切換 / (C) 重新派任務

## 6. 對應的失敗模式

當本條款被違反，依 `failure-modes.md` 對應如下（不另立新 F-mode）：

| 違反方式 | 失敗模式 |
|---|---|
| 衝突期間單方逕行（違反 §5.1 pending）| F5（規則記憶失效）+ 角色越界（連動 `role-separation.md`）|
| 衝突未留紀錄、私下口頭解決 | F4（編號 / 紀錄偏差類）|
| L2 上報缺立場 / 證據 / 選項 | F3（捏造數據類，因為「無理據主張」即捏造）|
| 把失敗事件包裝為分歧逃避 escalation | F1（假宣告）|
| 時效性例外未補紀錄 | F4 + F5 雙觸發 |

## 7. 與其他 core 條款的關係

| 條款 | 關係 |
|---|---|
| `escalation-protocol.md` | **嚴格區隔**（§0）；L1 仲裁若判定為失敗事件即轉 escalation 處置 |
| `role-separation.md` | §5.1 pending 紀律與 §3.1 越界退稿銜接；衝突期間越界即視為越界 |
| `evidence-first.md` | L0/L1 立場須附證據；無證據主張視為違反本條 |
| `audit-rights.md` | 衝突期間抽驗權不停止；對方提交的非衝突點仍須抽驗 |
| `violation-reflection.md` | 衝突結束後**不**強制反省（無對錯）；但 L2 級衝突若涉及條款解讀盲區，建議入反省檔強化條款 |
| `common-memory-root.md` | conflict-record 與 IM 判例放在 `<common-memory-root>/conflicts/` 與 `institutional-memory/` |
| `handoff-chain.md` | 跨 session 未解決的衝突須在 HANDOFF 標明 pending 狀態，接班 AI 不得逕行 |
| `cross-ai-handoff.md` | 跨 AI 接班時 pending 衝突須列入「未結案膠囊清單」（依 cross-ai-handoff §3.1）|
| `domain-axiom-slot.md` | §2「領域 vs 通用衝突」優先序的理論依據 |
