# Completion Delivery Contract（完工交付契約）

> **狀態**：v0.1
> **位階**：core 通用條款。
> **保證強度**：結構強制（核心 VCP；directive header 等格式項屬單 actor 自律輔助）
> **檢測時點**：runtime
> **since**：v0.1

---

## 1. 條文

每個執行角色（典型：Engineer）每完成一項任務（任意大小，含 hotfix），交付摘要的最後一節必須附上 **驗收測試計畫**（Acceptance Test Plan / VCP — Verification Checkpoint Plan）。

「全綠 / 通過」結論**不得單獨交付**；必須附驗收方可獨立執行的路徑。

---

## 2. 動機

驗收方（典型：PM）通常無法看到實作細節、不修改程式碼、可能無 shell 操作權，只能透過診斷工具 / UI / 行為觀察進行驗收。

執行角色對運行行為的掌握度高於驗收方，**翻譯成驗收方可執行的測試步驟是執行角色的責任**。

---

## 3. 必含結構（每節缺一不可）

### 3.1 條款 0 — Directive Header（驗收觸發信號）

明確要求驗收方接手執行，而非「接收」資訊。範本：

```markdown
> 📨 **致 <角色名稱>**：以下測試計畫由 <執行角色> 交付，請接手執行 ——
> 我的執行環境無法跨 session / 跨機驗證以下情境。請依序執行情境 1 → N，
> 把每段終端機輸出**原文**貼回此對話。我會判讀並決定是否關閉本任務。
> 可選情境（標 🟡）失敗不阻擋驗收。
```

**位階**：必置於計畫第一塊。

### 3.2 前置檢查（Pre-flight）

列出測試開跑前必須確認的環境 / 資料狀態。**用具體指令驗證**，禁止只寫文字描述。

### 3.3 雙保險檢查（領域安全相關時必含）

涉及不可逆動作（金融下單、生產資料寫入、外部副作用）的情境必含**雙重環境檢查**。範例（CryptoBot）：

- Pre-flight 第 1 段驗 `Mode=Demo`
- 情境內第 2 次再驗 `Mode=Demo`

避免驗收方在錯誤環境跑驗。各專案應在 `domain-axioms.md` 定義對應的雙保險規則。

### 3.4 測試情境（建議 3-5 個）

每個情境必含：

| 元素 | 必含 | 說明 |
|---|---|---|
| **目的** | ✅ | 本情境驗證的具體行為 |
| **指令** | ✅ | 可一鍵複製貼上的純粹代碼區塊（**嚴禁加行號**）|
| **危險度標籤** | ✅ | 📖 唯讀 / ⚠️ 會建立紀錄 / 🔥 會動到資金（或對應領域標）|
| **期望輸出錨點** | ✅ | 圈出關鍵字串 / 狀態碼錨點，**不接受「正常」這種模糊描述** |
| **失敗解讀表** | ✅ | 表格列出「看到 X → 代表 Y → 修法 Z」，讓驗收方初步分流 |

### 3.5 UI / 視覺檢核（若有 UI 層變更）

診斷工具涵蓋不了的部分（畫面顯示、廣播事件、狀態變化）必須寫明操作步驟與期望畫面元素，標註「非阻塞 / 觀察項」與「核心驗收項」的差別。

### 3.6 整體驗收判定

明確列出「全綠 → 可關任務」「哪幾條紅 → 對應到哪個風險」的決策表。

---

## 4. 強制原則

| 原則 | 說明 |
|---|---|
| 不可逆動作雙保險 | 領域內不可逆操作的情境**必須**前置檢查 + 情境步驟內各驗一次 |
| 指令格式 | 純粹代碼區塊，嚴禁行號，確保一鍵複製可用 |
| 不要包山包海 | 每個情境須能在 5 分鐘內判定綠 / 紅；過大的驗收計畫應拆任務 |
| 模糊描述禁令 | 嚴禁「優雅處理」「應該可以」「正常」等不可驗描述 |
