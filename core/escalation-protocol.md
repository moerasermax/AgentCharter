# Escalation Protocol（升級協議）

> **狀態**：v0.1
> **位階**：core 通用條款。

---

## 1. 三段式升級

| 階段 | 觸發條件 | 抽驗方應對 |
|---|---|---|
| **正常** | 偶發單次偏差，宣告方迅速補強 | 標準退稿 → 補強 → 重抽驗 |
| **強化抽驗模式** | 同事件累計同類偏差 ≥ 2 次（依 `failure-modes.md`）| 後續所有宣告須附 stdout 原文（`ls -la` / `git log` / probe 結果），不再受理純文字描述 |
| **結構性失靈** | 同事件累計 ≥ 3 次同類偏差，退稿循環失效 | **觸發使用者裁決**，抽驗方暫停接受該方任何結案宣告 |

---

## 2. 強化抽驗模式 — 抽驗方行為

進入後：

1. 抽驗方對所有宣告**強制要求**對應實證指令的 stdout 原文
2. 宣告方的「自查 Checklist」（見 §3）每次必附，缺項即視同未交付，工程師有權直接退稿不進入抽驗
3. 對應角色的歷史紀錄（capsule / handoff）須附「強化抽驗模式期間」標籤

---

## 3. 自查 Checklist（強化抽驗模式必附）

宣告方每次交付前須逐項打勾並附實證指令：

- [ ] 文件 / 變更放在約定位置（以 `ls -la <path>` 證實實體存在）
- [ ] 所有宣稱建立 / 修改的檔案都實際存在（每條獨立跑 `ls -la`，不接受批次描述）
- [ ] 所有引用的 git commit hash 都跑 `git log --oneline -1 <hash>` 證實
- [ ] 數值 / 統計類宣告附原始 query / probe / log 行
- [ ] 沒有「優雅處理 / 正常運作 / 應該可以」這類模糊描述
- [ ] 沒有未經實測就寫入的效能 / 延遲 / 規模數據

---

## 4. 使用者裁決例外（結構性失靈時）

當退稿循環失效，抽驗方**不得繼續單方面退稿到對方就範**，須上報使用者並附以下選項：

| 選項 | 行動 |
|---|---|
| **A** | 命令宣告方重做並附 commit hash / `ls -la` 等實證原文 |
| **B** | 暫時授權對方代寫宣告方的工作（單次例外，不形成慣例）|
| **C** | 跳過協議流程，直接由執行方完成（抽驗責任歸使用者）|
| **D** | 其他自由指令 |

使用者裁決後，抽驗方依指令執行；若選 B / C 須在歷史紀錄中**透明標註**「使用者裁決例外授權」。

---

## 5. 解除升級狀態

「強化抽驗模式」不會自動解除。解除條件：

1. 對應宣告方在後續 N 次（建議 N ≥ 3）連續綠燈無偏差
2. 抽驗方明示「解除強化抽驗」並寫入歷史紀錄
3. 跨 session 接班時，新抽驗方須讀過 `failure-modes.md` 的事件累積紀錄判斷是否延續

---

## 6. 與其他 core 條款的關係

| 條款 | 關係 |
|---|---|
| `audit-rights.md` | 升級的前提是抽驗權生效 |
| `failure-modes.md` | 升級條件由 F-catalog 累計觸發 |
| `role-separation.md` | 升級不得改變角色權限分配（除非使用者裁決選 B/C 明示）|
